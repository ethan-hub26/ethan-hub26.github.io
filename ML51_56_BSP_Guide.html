ML51 / ML56
ADVANCED GUIDE

NuMicro® 8051 패밀리 기반 개발자를 위한 BSP 구조, 프로젝트 빌드, WDT 리셋, SPROM 보안 메모리 전 항목 기술 가이드입니다. Keil uVision4 / PK51 V9.52, IAR EW8051, NuEclipse SDCC 세 툴체인을 모두 포함합니다.

지원 칩 및 BSP 개요

ML51 시리즈는 Nuvoton NuMicro® 8051 패밀리의 1T 8비트 마이크로컨트롤러입니다. BSP V2.00.001 기준으로 ML56, ML54, ML51(64KB/32KB/16KB Flash) 네 가지 변형을 지원합니다.

KEIL PK51 프로젝트 설정

💡
권장 방법

신규 프로젝트는 Template\Project_temp\Keil\ 폴더를 복사하여 시작하는 것이 새로 생성하는 것보다 훨씬 안정적입니다.

프로젝트 생성 절차

1
템플릿 폴더 복사

Template\Project_temp\ 폴더 전체를 동일 레벨에 복사합니다. (예: Template\MyProject\)

2
프로젝트 파일 이름 변경

복사된 폴더 내의 project_temp.uvproj를 원하는 이름(예: MyProject.uvproj)으로 변경합니다.

3
KEIL에서 프로젝트 이름 변경

KEIL 상단 Target 드롭다운에서 프로젝트 이름을 수정합니다. Options → Output → Name of Executable도 함께 변경합니다.

4
빌드 및 출력 파일 확인

빌드 성공 시 Output\ 폴더에 .hex와 .bin 파일이 생성됩니다.

Options for Target — 탭별 설정

Device 탭

Vendor: Nuvoton → Device: 사용하는 칩(예: MS51FB9AE) 선택. Use Extended Linker (LX51) 체크 권장.

⚠
Nuvoton 8051 Device 미표시 시

Keil Device Family Pack(DFP)이 설치되지 않은 경우입니다. Nuvoton 공식 사이트에서 Nuvoton NuMicro 8051 DFP를 다운로드하여 설치하십시오.

Target 탭

Memory Model

Large: variables in XDATA 선택 필수. 기본 변수가 XRAM에 배치됩니다.

Code Rom Size

Large: 64K program. Flash 크기에 맞게 선택합니다.

Operating system

None (RTOS 미사용 시)

Xtal (MHz)

24.0MHz 권장 (HIRC 기준). 실제 동작 클럭과 일치시켜야 UART 보레이트가 정확합니다.

Output 탭

Name of Executable: 출력 .hex 파일명 결정. Keil은 HEX만 생성하며, BIN 파일은 Nuvoton 제공 hex2bin.exe로 변환합니다.

User 탭 — hex2bin 자동 변환

After Build/Rebuild → Run #1에 아래 커맨드를 등록하면 빌드 후 자동으로 BIN이 생성됩니다.

Keil > Options > User > After Build/Rebuild Run #1 CMD
# BIN 전체 변환 (기본 0xFF 패딩)
$KC51\NULink\Hex2Bin.exe .\output\@L.hex

# 범위 지정 변환: 시작주소 0x0000, 크기 0x4000 (16KB)
$KC51\NULink\Hex2Bin.exe .\output\@L.hex 0x0 0x4000

# APROM + SPROM 분리 추출 (SPROM 시작 0xFF80, 크기 0x80)
$KC51\NULink\Hex2Bin.exe .\output\@L.hex 0x0 0x1000 0xFF80 0x80

컴파일러별 조건부 컴파일 매크로

예시: numicro_8051.h 또는 common.cC
#if defined __C51__ // KEIL PK51
  uint8_t data TA_REG_TMP, BYTE_TMP, SFRS_TMP;
#elif defined __ICC8051__ // IAR EW8051
  uint8_t __data TA_REG_TMP, BYTE_TMP, SFRS_TMP;
#elif defined __SDCC__ // NuEclipse SDCC
  __data uint8_t TA_REG_TMP, BYTE_TMP, SFRS_TMP;
#endif

특정 툴체인만 사용한다면 다른 #elif 블록 전체를 제거하여 코드를 단순화할 수 있습니다.

WDT 리셋 안정화 — CONFIG 소프트웨어 수정 금지

🚫
양산 현장 필수 주의사항

소프트웨어(IAP)로 CONFIG를 수정하여 WDT를 활성화하는 방식은 양산 제품에서 반복 리셋 및 시스템 오류의 원인이 됩니다. 반드시 ICP 프로그래밍 단계에서 CONFIG를 설정하십시오.

WDT 기본 동작

ML51 시리즈 WDT는 기본값으로 범용 타이머로 설정됩니다. 주기적 인터럽트는 이벤트 타이머나 시스템 감시에 사용할 수 있으며, Idle/Power-down 모드에서도 동작 가능합니다. 그러나 기본값에서는 리셋 기능이 없습니다.

리셋 기능을 사용하려면 CONFIG 바이트의 WDTEN[3:0] (CONFIG4[7:4])을 설정해야 합니다. (TRM 6.1.4 Config bytes 참조)

CONFIG 설정값 — WDT Options

옵션CONFIG4[7:4]동작권장
Disabled1111WDT 완전 비활성화—
Enabled, stop in Idle/PD0101WDT 리셋 활성화, Idle/Power-down에서 정지—
Enabled, keep running in Idle/PD0000WDT 리셋 활성화, Idle/Power-down에서도 계속 동작권장

소프트웨어 CONFIG 수정을 피해야 하는 이유 (4가지)

①
리셋 후에야 적용

소프트웨어로 CONFIG를 변경해도 리셋 이후에만 유효합니다. 따라서 모든 리셋 경로(POR, 리셋 핀, BOD 리셋, WDT 리셋)에서 중복 수정 방지 로직을 별도로 구현해야 합니다.

②
전원 노이즈 취약성

실제 양산 환경에서는 전원이 요동치거나 EFT(Electrical Fast Transient) 간섭이 발생합니다. IC가 리셋된 후 수십~수백 ms 만에 재리셋되는 상황이 반복될 수 있습니다.

③
CONFIG 재기록 도중 리셋 위험

CONFIG 재기록 과정은 ① 원본 저장 → ② 새 데이터 수정 → ③ CONFIG 영역 소거 → ④ 재기록의 4단계입니다. ①②③ 중 어느 단계에서 리셋이 발생해도 CONFIG가 불완전한 상태로 남아 부팅 실패를 유발합니다.

④
ICP 1회 기록 vs. 소프트웨어 반복 기록

ICP로 기록한 CONFIG는 의도치 않은 다른 동작으로 절대 변경되지 않습니다. 반면 소프트웨어로 기록하면 시스템 운용 중 반복 기록이 가능해 Flash 내구성 및 시스템 안정성이 저하됩니다.

🚫
결론

Nuvoton FAE 현장 사례에서 소프트웨어 CONFIG 재기록 방식은 양산 제품에서 반복적인 시스템 오류를 유발함이 확인되었습니다. WDT 활성화는 반드시 ICP 프로그래밍 단계의 CONFIG 설정으로만 수행하십시오.

08

SPROM — 보안 보호 메모리

MS51 / ML51 시리즈는 128바이트 보안 보호 메모리(SPROM)를 내장합니다. 암호화 키, 인증 코드, 보안 부트 루틴 등 보호가 필요한 코드를 저장하는 데 사용합니다.

SPROM 메모리 맵

FFFFH
SPROM 잠금 비트 — 0xFF: 비보안 모드 / 기타: 보안 모드
FF80H~FFFEH
SPROM 코드 영역 (127 bytes)
3C00H~3FFFH
LDROM 영역 (CONFIG 설정에 따라 크기 결정)
0000H~3BFFH
APROM 영역 (최대 15KB, 16KB Flash 기준)
⚠
SPROM 특성

SPROM은 "Whole Chip Erase" 명령을 지원하지 않습니다. 별도 소거/프로그램 절차가 필요합니다.

보안 모드 vs 비보안 모드

모드FFFFH 값CPU 읽기IAP/ICP 읽기소거/프로그램OCD 브레이크
비보안 0xFF 가능가능가능가능
보안 0xFF 아님 SPROM 영역 내 실행만 가능 (외부 접근 시 0x00 반환) 불가불가불가

Keil 프로젝트에서 SPROM 코딩 방법

1단계 — APROM main에서 SPROM 함수 호출

main.cC
#include "numicro_8051.h"

void main(void)
{
  MODIFY_HIRC(HIRC_24);
  set_IAPUEN_SPMEM; // SPROM 활성화
  ALL_GPIO_QUASI_MODE;
  UART_Open(24000000, UART0_Timer1, 115200);
  ENABLE_UART0_PRINTF;
  while(1)
  {
    SPROM_CODE(); // SPROM 내 함수 호출
    printf("\n\rSPTEMP= 0x%BX", SPTEMP);
  }
}

2단계 — SPROM.c 작성

SPROM.cC
#include "MS51.h"
#include "SPROM.h"

unsigned char SPTEMP = 1;

void SPROM_CODE(void)
{
  P12 = ~P12; // GPIO 토글
  SPTEMP++; // 카운터 증가
}

3단계 — ProtectBit.c — SPROM 잠금 비트 제어

ProtectBit.cC
#include "MS51.h"

// 0xFF: 잠금 해제 (비보안 모드)
// 0x00: 잠금 (보안 모드 — 이후 IAP/ICP로 읽기 불가)
unsigned char code protect[] = {
  0xFF // 개발 중: 0xFF (비보안)
  // 0x00 // 양산 시: 0x00 (보안 잠금)
};

4단계 — Keil BL51 Locate 탭 — SPROM 주소 고정

Options → BL51 Locate → Code 란에 아래 내용을 입력합니다:

Options > BL51 Locate > Code (Segments)LINKER
?PR?SPROM_CODE?SPROM(0xFF80), ?CO?PROTECTBIT(0xFFFF)

이 설정으로 SPROM 코드가 0xFF80에, 잠금 비트가 0xFFFF에 배치됩니다.

ℹ
Macro 이름 확인 방법

BL51 Locate의 Code 섹션에 입력할 Macro 이름은 Project\Lst\project.m51 파일에서 확인합니다. 예: 아래와 같이 ?PR?SPROM_CODE?SPROM이 FF80H에 배치된 것을 확인 가능합니다.

APROM + SPROM BIN 파일 분리 생성

ICP 툴로 개별 프로그래밍 하거나 양산용 BIN을 분리할 때 사용합니다.

# Run #1: APROM 전체 BIN 추출 (0x0000 ~ 0x0FFF, 4KB)

# Run #2: APROM + SPROM 분리 BIN 생성
# 인자: APROM시작 APROM크기 SPROM시작 SPROM크기
..\..\..\Library\Startup\exe\hex2bin.exe .\output\@L.hex 0x0 0x1000 0xFF80 0x80
